# MVP & Idea Validation

The point of an MVP is **maximum learning for minimum build** — validate that people want this before investing in building it well. Most ideas are wrong in some way; find out cheaply.

## Start From the Riskiest Assumption

Every idea rests on assumptions (people have this problem, they'll pay, they'll switch, we can reach them, we can deliver it). List them and **test the one that, if false, kills the idea** — usually a demand/value assumption, not a technical one. Don't build the easy part first; test the scariest belief first.

## Smallest Experiment That Gives a Real Signal

Match the cheapest test to the assumption:
- **Landing page / smoke test** — describe the offer, measure sign-ups/pre-orders. Tests demand without building anything.
- **Concierge** — deliver the value manually, by hand, for a few real customers. Tests whether the value lands before automating it.
- **Wizard of Oz** — the front looks automated; humans do the work behind the curtain. Tests the experience without the engineering.
- **Prototype / clickable mockup** — tests usability and comprehension.
- **Interviews** — test the *problem's* existence and severity (but watch: people are polite; behavior beats stated intent).

An MVP is not "version 1 with fewer features" — it's an experiment. It can be a page, a spreadsheet, or a manual service.

## Decide in Advance, Then Honor It

Before running the test, write down **what result means go and what means stop** (e.g. "≥X% of visitors pre-order"). Deciding the bar afterward invites rationalizing a failure into a success. Prefer behavioral signals (someone paid, signed up, kept using it) over opinions ("that's cool, I'd totally use it").

## Persevere, Pivot, or Kill

From the evidence: **persevere** (signal is there — build more), **pivot** (the problem is real but your solution/segment is wrong — change one big thing), or **kill** (no real demand — stop, and that's a win: you saved months). Falling in love with the idea and ignoring the signal is the core failure mode. Iterate: each validated assumption unlocks testing the next one.

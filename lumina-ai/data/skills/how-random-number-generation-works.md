# How Random Number Generation Works

Computers are deterministic, so "random" numbers are usually **pseudo-random** — generated by an algorithm that *looks* random but is fully determined by a starting **seed**. Knowing the difference between pseudo-random and cryptographically-secure randomness is essential, because getting it wrong breaks security.

## Pseudo-Random Generators (PRNGs)

A **PRNG** starts from a **seed** and applies a deterministic formula to produce a sequence that passes statistical tests for randomness (uniform, unpredictable-looking). Key facts:
- **Deterministic** — the same seed always produces the **same sequence**. This is a *feature* for **reproducibility**: seed a simulation/test/shuffle to replay it exactly (crucial for debugging and scientific reproducibility).
- **Not truly random** — it's a fixed sequence; if you know the algorithm and seed (or enough outputs), you can predict the rest.
Common PRNGs (Mersenne Twister, PCG, xorshift) are fast and statistically excellent — great for **simulations, games, sampling, tests** — but **must not** be used where an adversary matters.

## Cryptographically Secure RNGs (CSPRNGs)

For anything security-sensitive — **passwords, tokens, session IDs, keys, salts, nonces** — you need a **CSPRNG**: a generator whose output is **unpredictable even to an attacker** who sees previous outputs, and whose internal state can't be reverse-engineered. CSPRNGs are seeded from real **entropy** (below) and designed so that observing outputs reveals nothing about future ones. Use the OS/crypto library: `secrets` (Python), `crypto.randomBytes` (Node), `/dev/urandom`, `SecureRandom` — **never** `random`/`Math.random` for these.

## Entropy — the real randomness

To seed unpredictably, systems collect **entropy** — genuine physical unpredictability: hardware noise, timing of interrupts/mouse/disk, dedicated hardware RNGs (Intel RDRAND). The OS pools this entropy and feeds CSPRNGs. True randomness comes from the physical world; the algorithm just stretches it into a stream. (Freshly-booted VMs/embedded devices can be **entropy-starved** — a real vulnerability source.)

## The Critical Distinction

- **PRNG** (fast, reproducible, predictable) → simulations, games, shuffling, sampling, tests.
- **CSPRNG** (unpredictable, secure) → tokens, keys, passwords, salts, anything security-relevant.
Using a plain PRNG for security is a **serious vulnerability** — attackers have broken systems by predicting "random" session tokens or crypto keys generated with `Math.random` or a known/weak seed.

## Pitfalls (in understanding/using)

- Using **`Math.random`/`rand`/`random`** for tokens, keys, passwords, or anything secret → predictable, exploitable. Use a **CSPRNG**.
- **Seeding a CSPRNG with something predictable** (time, PID) → defeats it; let the OS seed from entropy.
- Expecting **reproducibility** from a CSPRNG (you don't control the seed) — use a seeded PRNG when you need repeatable sequences.
- **Entropy starvation** on boot/VMs/embedded → weak early keys; ensure sufficient entropy.
- Assuming "passes statistical tests" = "secure" — statistical randomness ≠ cryptographic unpredictability.
- Reusing nonces/tokens or truncating CSPRNG output carelessly (weakens guarantees).
